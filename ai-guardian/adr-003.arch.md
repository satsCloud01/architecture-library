# ADR-003: LLM-Specific Governance Extension

**Status:** Accepted
**Date:** 2025-03-01
**Authors:** AI Risk Team, Platform Engineering

---

## Context

Enterprise AI Guardian must govern both classical ML/statistical models and LLMs/GenAI systems within a single platform. These two model types have fundamentally different risk profiles:

| Dimension | Classical ML | LLM / GenAI |
|---|---|---|
| Output type | Deterministic score / class | Generative text (non-deterministic) |
| Key failure mode | Data drift, bias | Hallucination, prompt injection, harmful output |
| Performance metric | AUC, PSI, Gini, F1 | Hallucination rate, relevance, latency |
| Regulatory coverage | SR 11-7, ECOA, Basel III | EU AI Act, SR 11-7 (extended), DORA |
| Vendor dependency | Internal / open-source | OpenAI, Anthropic (third-party) |
| Cost structure | Compute (fixed) | Token-based (variable) |
| Explainability | SHAP / LIME available | Often black-box; RAG provides grounding |

Most existing model risk management frameworks (including SR 11-7) were designed before LLMs existed and require interpretation.

---

## Decision

LLM-specific governance fields are added directly to the `AIModel` entity as nullable columns. They are only populated and displayed when `model_type = LLM`.

### Added Fields

| Field | Type | Governance Purpose |
|---|---|---|
| `prompt_injection_risk` | `Low/Medium/High` | Security control — adversarial input exposure |
| `hallucination_rate` | `float %` | Output quality monitoring — factual accuracy |
| `output_filtering` | `boolean` | Control presence — PII/harmful content filtering |
| `rag_enabled` | `boolean` | Architecture flag — grounding approach |
| `human_in_loop_required` | `boolean` | Oversight control — human review requirement |
| `monthly_token_cost_usd` | `float` | Vendor spend tracking — third-party dependency |

### Performance Metrics Extension

LLM performance metrics tracked separately from ML metrics in `performance_metrics`:
- `hallucination_rate` — monthly trend
- `response_relevance` — RAG quality score
- `latency_p95_ms` — operational SLA
- `token_cost_usd` — spend management

### EU AI Act Mapping

LLMs deployed in high-risk use cases (credit, employment, biometrics) are classified as **high-risk AI systems** under EU AI Act Article 6. The compliance mapping to `EU-AI-ACT` captures the 6 required controls:
1. Technical documentation (Annex IV)
2. Human oversight mechanism
3. Transparency requirements
4. Conformity assessment
5. Post-market monitoring plan
6. High-risk classification documented

---

## Consequences

**Positive:**
- Single platform governs the full AI portfolio without siloing LLMs
- Differentiates the platform from legacy MRM tools (which ignore LLMs)
- EU AI Act coverage is a competitive advantage as EU operations expand

**Negative:**
- Nullable columns for LLM fields add complexity to the `AIModel` schema
- Hallucination rate measurement requires integration with output evaluation tooling (not yet implemented — tracked as metric in demo data)
- Vendor dependency risk (OpenAI/Anthropic API changes) is captured but not yet actively monitored via webhook

## Future Considerations

- **Prompt library governance** — track prompt templates as versioned artefacts
- **Red team testing** — formal adversarial testing records as validation controls
- **Model cards (Hugging Face standard)** — export model cards in structured format
- **LLM-specific SLAs** — p99 latency, max hallucination threshold SLA agreements with providers
