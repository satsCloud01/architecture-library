# ADR-002 — ABAC Policy Model Design

**Status:** Accepted
**Date:** 2025-01
**Author:** Platform Team

---

## Context

The system must make fine-grained access control decisions for a heterogeneous set of requestors (humans, ETL pipelines, AI agents) accessing multiple data platforms (Snowflake, PostgreSQL, Databricks, S3, MySQL).

Access decisions must consider:
- **Who** is requesting (role, region, clearance level, principal type)
- **What** they are accessing (datasource, tables, columns, geographic regions)
- **What operation** they want to perform (SELECT, INSERT, UPDATE, DELETE)
- **Contextual factors** (time of day, IP range, stated purpose)

The system must also support **obligations** — conditional constraints applied with a PERMIT decision, such as masking PII columns or filtering rows by region.

We evaluated: RBAC, ABAC, ReBAC (relationship-based), and OPA (external policy agent).

---

## Decision

### ABAC (Attribute-Based Access Control)

PolicyGuard implements a native ABAC engine with the following policy structure:

```json
{
  "name": "PII Read - EU Data Scientists Only",
  "effect": "permit",
  "state": "active",
  "subject_json": {
    "role": ["data_scientist"],
    "region": ["EU"],
    "clearance_level": {"gte": 2}
  },
  "resource_json": {
    "datasource": ["snowflake_prod"],
    "tables": ["customers", "accounts"],
    "columns": {
      "allow": ["customer_id", "name", "email"],
      "masked": ["ssn", "dob"]
    },
    "row_filter": "data_region = 'EU'"
  },
  "action_json": ["SELECT"],
  "conditions_json": {
    "time_window": {"start_hour": 8, "end_hour": 20},
    "max_rows_per_query": 10000
  }
}
```

### Combining Algorithms

Three algorithms are supported, configurable at runtime via Settings:

| Algorithm | Semantics | When to Use |
|---|---|---|
| `deny-overrides` | Any DENY match → final DENY | Default; maximally restrictive; compliance-first |
| `permit-overrides` | Any PERMIT match → final PERMIT | Permissive environments; exception-heavy access patterns |
| `first-applicable` | First matched policy (by priority) wins | Ordered rule chains; legacy policy migration |

### Policy Versioning

Every mutation of a policy appends a `PolicyVersion` snapshot to an append-only version table. This enables:
- Full audit trail of who changed what and when
- Rollback by re-applying a prior version's JSON
- Diff comparison in Policy Explorer

### Policy State Machine

```
draft → review → active → deprecated → archived
  ↑        ↓
  └────────┘
```

- `draft`: authored but not evaluated
- `review`: submitted for compliance sign-off
- `active`: evaluated in all PDP calls
- `deprecated`: not evaluated; retained for audit history
- `archived`: soft-deleted; invisible to PDP

---

## Consequences

**Positive:**
- A single `policies` table can express any access control rule without schema changes
- Obligations (masking, row filtering) are co-located with the permit rule — no separate enforcement table
- Combining algorithm is a runtime config, not a code change — operational flexibility
- ABAC naturally handles AI Agent principals via `principal_type` attribute (no special code path)

**Negative / Trade-offs:**
- JSON columns (`subject_json`, `resource_json`, etc.) are not indexed — policy matching is a full table scan of active policies; acceptable for < 500 active policies; requires indexing or caching at > 10K policies
- No XACML compliance — a deliberate choice to keep the engine readable and testable; XACML import/export is a future enhancement
- Combining algorithm is global, not per-policy-set — can be extended to per-category in a future ADR

---

## Alternatives Considered

### RBAC (Role-Based Access Control)

RBAC assigns permissions to roles. Rejected because:
- Cannot express attribute-based constraints (region, clearance level, time window)
- Cannot express obligations (masked columns, row filters)
- No path to AI Agent principal type differentiation

### OPA (Open Policy Agent)

OPA is a mature ABAC engine with Rego policy language. Rejected for Phase 1 because:
- Adds an external process dependency (OPA sidecar)
- Rego is a DSL unfamiliar to most teams; harder to generate from Claude AI suggestions
- PolicyGuard's engine is simpler to audit, test, and extend for demo purposes
- OPA migration is viable in Phase 2 if policy volume exceeds 10K rules

### ReBAC (Relationship-Based Access Control)

ReBAC (Zanzibar-style) models permissions as graph relationships. Rejected because:
- Optimised for user-resource relationship hierarchies (e.g., Google Drive sharing)
- Overkill for data platform access control which is attribute-driven, not relationship-driven
